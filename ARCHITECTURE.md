# lastsave — Документация проекта

## Структура курса

Курс — это Java Core. Разбит на **блоки** (спринты). Каждый блок привязан к GitHub-шаблону (`template_repo_name`) — при старте блока студенту создаётся репозиторий из шаблона.

**Иерархия контента:** курс → блок → топик → секция. Секция — минимальная единица прогресса. Паттерн: в каждом топике две секции — «Теория» и «Задание».

**Разблокировка секций** (`unlock_condition`):
- `MANUAL` — секция считается пройденной после нажатия кнопки «Готово» в Telegram
- `PR_REQUIRED` — секция ждёт смерджженного PR в GitHub

**Сообщения** — это скрипт диалога. Каждое сообщение имеет:
- `sender` (`ANYA` / `BOT`) — кто пишет
- `wait_for` (`NOTHING` / `CALLBACK` / `PR`) — что ждём перед следующим шагом
- `callback_text` — текст кнопки в Telegram (если `wait_for = CALLBACK`)

**Прогресс студента** фиксируется в `student_progress` — одна запись на текущую/завершённую секцию.

**message_log** — аудит отправленных сообщений. `callback_received_at` — момент нажатия кнопки, `pr_received_at` — момент получения PR.

## Сценарий от начала до конца

1. Студент пишет `/start` в бот → авторизуется через GitHub OAuth → запись в `students`
2. После регистрации вызывается `startProgress(student)` → находит первый блок → первый топик → первую секцию → создаёт запись в `student_progress`, публикует `BlockStartedEvent`
3. `BlockStartedEvent` обрабатывается хендлером: создаётся репозиторий из шаблона, студенту отправляется уведомление со ссылкой
4. Шедулер видит студента с активным прогрессом и начинает слать сообщения секции, записывая в `message_log`
5. Когда все сообщения секции отправлены — шедулер останавливается и ждёт внешнего события
6. Студент нажимает кнопку (`CALLBACK`) → хендлер завершает секцию, открывает следующую; или приходит смерджженный PR → PR-хендлер делает то же самое
7. Если следующей секции нет — вызывается `startBlock()` для следующего блока → `BlockStartedEvent` → новый репозиторий

## Логика шедулера

Шедулер запускается каждую минуту. Его единственная задача — отправить следующее сообщение студенту, если оно готово к отправке. Шедулер не двигает прогресс напрямую, не трогает GitHub, не инициализирует первый блок.

**Один тик = одно действие на студента.**

### Шаги за один тик

**1.** Найти активный прогресс студента (`IN_PROGRESS`). Нет прогресса — пропустить.

**2.** Найти последний лог сообщения для текущей секции.

**3.** Если логов нет — секция новая. Отправить первое сообщение + материалы секции. Тик закончен.

**4.** Проверить `wait_for` последнего сообщения:
- `NOTHING` → готово, идём дальше
- `CALLBACK` → проверяем `callback_received_at`. Если null — ждём. Тик закончен.
- `PR` → проверяем `pr_received_at`. Если null — ждём. Тик закончен.

**5.** Найти следующее сообщение секции. Если есть — отправить. Тик закончен.

**6.** Если следующего нет — секция завершена. Вызвать `completeSectionAndAdvance` → отправить первое сообщение новой секции + материалы. Тик закончен.

## AI ревью PR

При создании PR в репозитории студента срабатывает GitHub Actions:
1. Получает diff PR
2. Отправляет POST на `/api/pr-review` с `github_username`, `pr_url`, `diff`
3. Эндпоинт защищён `X-Secret-Token`

Сервис при получении запроса:
1. Находит студента по `github_username`
2. Помечает `pr_received_at` в `message_log` — шедулер продвинет прогресс на следующем тике
3. Отправляет diff в OpenAI, получает ревью
4. Отправляет ревью студенту в Telegram (разбивает на части по 2000 символов)

## Ключевые сервисы

| Сервис | Ответственность |
|--------|----------------|
| `StudentProgressService` | Старт прогресса, старт блока, завершение секции и переход к следующей |
| `ProgressFlowService` | Оркестрация одного тика шедулера |
| `MessageLogRepository` | Логирование отправленных сообщений, фиксация callback/PR |
| `NotificationService` | Отправка сообщений в Telegram |
| `AiReviewService` | Отправка diff в OpenAI, получение ревью |
| `PrReviewService` | Обработка входящего PR-ревью: markPrReceived + отправка ревью студенту |
